### Epic summary

Epic 12 delivered **data-sovereignty and version UX** on top of the collaborative stack from earlier epics: a **GDPR profile center** (consents, export, deletion workflow), **Ghost Mode** visual diff with **Web Worker** offload, and **secure snapshot restore** with **Redis locking**, **Hocuspocus stateless signals**, **HTTP 423** while locked, and **SHA-256** integrity checks.

---

### What went well

- **Consistent cross-cutting pattern:** Redis appears for **export/deletion concurrency** (12.1) and **restore locking** (12.3), aligning with existing platform use (tickets, caches).
- **UX + compliance in one epic:** Profile center and dangerous actions (export, delete, restore) follow **Azure Flow** and explicit confirmation patterns.
- **Performance isolation:** Ghost Mode diff in a **Web Worker** (12.2) matches NFR expectations for long documents without blocking the main thread.
- **Restore safety:** Centralized restore core with **transactional DB update**, **integrity check**, and **review-driven test expansion** (canonical path + 502 on hash mismatch) in 12.3.

---

### Challenges & system-level insights

- **Collaboration edge cases:** 12.3 review noted that if restore **fails after lock**, remote clients may still see **unlock** without a dedicated **`document_restore_failed`** broadcast — **deferred** as a product enhancement, not a blocker.
- **Stack coordination:** Restore touches **FastAPI + MinIO + Postgres + Redis + Hocuspocus**; the epic succeeded by **one core restore path** and clear ordering (lock → signal → swap → reload → unlock in `finally`).
- **Story 12.1** stories are specification-dense (JWT, streaming ZIP, anonymization); **early alignment** on Keycloak `sub` and audit tables avoided rework.
