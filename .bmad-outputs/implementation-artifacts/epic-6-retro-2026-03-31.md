## 3. What went well

- **State machine rigor:** Explicit lifecycle boundaries prevented bypasses (`PUT /status` cannot directly complete project).
- **Race-safety improvements:** Conditional SQL updates and idempotent branches reduced duplicate side effects.
- **Resilient orchestration boundary:** Camunda failures are logged and tolerated where contract requires API closure to remain successful.
- **Review quality:** Multiple findings were translated into concrete patches, not deferred as "later debt".
- **Documentation parity:** `docs/api-mapping.md`, story records, and tests were kept synchronized with implementation behavior.

## 4. Challenges and debt signals

| Theme | Evidence in Epic 6 | Debt / risk signal |
|------|---------------------|--------------------|
| Test-model drift | Mock expectations initially diverged from hardened endpoint behavior | Add shared test builders for refreshed/concurrent state paths |
| Late edge hardening | Several critical issues surfaced in review phase | Shift adversarial checks earlier in story execution |
| Runtime environment coupling | Missing dependencies in active Python interpreter slowed initial validation | Add one-command local backend test bootstrap/verification |
| Contract precision burden | Error payload shape assumptions caused failures | Enforce API contract assertions with reusable helpers |

## 5. Quality insights

- The most valuable tests in this epic were **authorization + lifecycle + idempotency + failure-tolerance** combinations, not isolated happy paths.
- Privacy-safe logging improvements (metadata instead of raw rejection content) should be maintained as a default rule.
- Camunda boundary behavior is now clearer: trigger is best-effort, closure transition is durable first.

- Preserve strict lifecycle transitions and reject bypass paths explicitly.
